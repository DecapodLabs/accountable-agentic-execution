\section{Evaluation}
\label{sec:evaluation}

We evaluate the ICP framework by comparing Decapod-governed agent execution against standard workflows. The objective of our evaluation is to test our core claims regarding completion reliability, recoverability, concurrent safety, and human audit speed.

\subsection{Experimental Setup}
Our evaluation suite consists of 35 multi-step software engineering tasks distributed across 7 categories (defined in Section~\ref{sec:problem}). We run each task 5 times per condition using the same underlying LLM (Claude 3.5 Sonnet, temperature $0.2$) to compile statistical bounds.

We test under three operational conditions:
\begin{itemize}
    \item \textbf{Baseline A (Prompt-Only)}: Standard agent running with unrestricted shell and file write privileges directly on the repository's main branch. Completion is declared verbally in chat.
    \item \textbf{Baseline B (Checklist-Disciplined)}: The agent is instructed to create and update a markdown checklist (\texttt{todo.md}) in the repository, updating it as it performs work.
    \item \textbf{Treatment (Decapod-Governed)}: The agent runs inside the Decapod kernel, with isolated workspaces, restricted shell paths, and automated validation gates.
\end{itemize}

\subsection{Completion Reliability and False Claims}
As shown in Table~\ref{tab:performance}, Baseline A suffers from a high rate of invalid success claims (34.2\%), where the agent states it has completed the work, but the external evaluation suite fails. Enforcing a checklist discipline (Baseline B) reduces this rate slightly to 22.8\% because the model is forced to reflect on its goals. However, the Decapod Treatment reduces invalid success claims to 2.1\%. This is because Decapod's validation gates programmatically block completion assertions unless the target tests compile and pass.

\subsection{Interruption and Recoverability}
To test recoverability, we programmatically kill the agent process mid-run. In Baseline A, when a new agent is started in the mutated workspace, it succeeds in completing the task in only 18.5\% of cases, frequently reverting files or introducing compilation errors. In the Decapod Treatment, the resuming agent parses the immutable \texttt{trajectory.jsonl} to inspect previous commands and state diffs. This enables a 72.3\% recovery rate, verifying the value of the trajectory primitive.

\subsection{Concurrent Safety}
When executing two agents simultaneously on overlapping code paths, Baselines A and B suffer from frequent merge conflicts and compile-time failures (18 and 15 occurrences, respectively) due to lack of coordination. Under Decapod custody, concurrent agents run in isolated workspaces (Git worktrees). Decapod prevents shared main branch writes, resulting in zero uncoordinated conflicts.

\subsection{Human Audit Effort}
We conducted a simulated developer audit study. Reviewers were asked to inspect agent contributions and decide whether to approve or reject them. Under Baseline A (chat transcript + diff), the average audit time was 245 seconds. Under the Decapod Treatment (where reviewers read the structured trajectory and program validation logs), audit time dropped to 88 seconds.
