\section{Evaluation}
\label{sec:evaluation}

We evaluate the proposed primitives by comparing Decapod-governed agent execution against standard workflows. The objective of our evaluation is to test our core claims regarding completion reliability, recoverability, concurrent safety, and human audit speed.

\subsection{Experimental Setup}
Our evaluation suite consists of 35 multi-step software engineering tasks distributed across 7 categories (defined in Section~\ref{sec:problem}). We plan to run each task 5 times per condition using the same underlying LLM (Claude 3.5 Sonnet, temperature $0.2$) to compile statistical bounds.

We test under three operational conditions:
\begin{itemize}
    \item \textbf{Baseline A (Prompt-Only)}: Standard agent running with unrestricted shell and file write privileges directly on the repository's main branch. Completion is declared verbally in chat.
    \item \textbf{Baseline B (Checklist-Disciplined)}: The agent is instructed to create and update a markdown checklist (\texttt{todo.md}) in the repository, updating it as it performs work.
    \item \textbf{Treatment (Decapod-Governed)}: The agent runs inside the Decapod kernel, with isolated workspaces, restricted shell paths, and automated validation gates.
\end{itemize}

\subsection{Completion Reliability and False Claims}
We plan to evaluate whether the validation primitive reduces invalid completion claims. Under Baseline A, we expect the agent to exhibit a high rate of invalid success claims (where the agent verbally asserts completion, but the external evaluation suite fails). Enforcing a checklist discipline (Baseline B) is projected to reduce this rate slightly because the model is forced to reflect on its goals. The Decapod Treatment is expected to minimize invalid success claims to near zero, as validation gates programmatically block completion assertions unless the target tests compile and pass.

\subsection{Interruption and Recoverability}
To test recoverability, our evaluation protocol programmatically interrupts the agent process mid-run and initiates a recovery run. In Baseline A, when a new agent is started in the mutated workspace without historical context, we expect low recovery rates, as the agent often struggles to reconstruct intent. In the Decapod Treatment, the resuming agent is designed to parse the immutable trajectory ledger to reconstruct state, which we hypothesize will significantly improve task completion rates post-interruption.

\subsection{Concurrent Safety}
In our concurrent execution tests, two agents attempt to make overlapping modifications. We expect Baselines A and B to suffer from frequent merge conflicts and corrupted repository states due to the absence of isolation. Under Decapod custody, concurrent agents run in isolated workspaces (Git worktrees), which we hypothesize will eliminate uncoordinated branch collisions and prevent concurrent write conflicts.

\subsection{Human Audit Effort}
We plan to conduct a user study measuring human developer audit time. Reviewers will be asked to approve or reject agent modifications under the different conditions. We hypothesize that reviewers presented with Decapod's structured trajectory and validation logs will confirm correctness significantly faster than those manually reviewing conversational transcripts and raw diffs under Baseline A.
