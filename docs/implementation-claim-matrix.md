The inspected Decapod revision is [`e7df80d8234a80c490f6fe2119a6cff32135a386`](https://github.com/DecapodLabs/decapod/commit/e7df80d8234a80c490f6fe2119a6cff32135a386), tagged `v0.66.1`, on 2026-07-12. ## Intentionally narrowed or rejected claims

- **Rejected:** “one active owner per task” as an unconditional invariant. The code supports trust-gated shared secondary owners; the paper instead specifies exclusive primary custody for the controlled studies.
- **Narrowed:** handoff continuity means durable task assignment, summary, events, knowledge, and available repository/proof state survive. It does not mean conversation-state transfer or lossless reasoning replay.
- **Narrowed:** provider independence means the durable governance contract and repository state are external to one provider conversation. It does not guarantee workbench compliance.
- **Narrowed:** trajectory is selected governance activity, not complete runtime interception or a universal tamper-proof ledger.
- **Narrowed:** proof portability means configured evidence can be inspected by a later agent; it does not make local evidence sufficient for integration publication.
- **Rejected as current capability:** remote/cloud fleet coordination. The public crate records an experimental opt-in payload but returns a backend-unavailable boundary.
- **Rejected as current capability:** organization-level constitution overlays. The implemented authority chain is embedded baseline plus project-local override.
- **Rejected as current capability:** standards adoption, native MCP/A2A support, HTTP governance transport, and vendor pre-bundling. The current thin waist is local CLI plus Decapod-specific process RPC; the standards profile and adapters are prospective.
- **Rejected as empirical result:** reduced duplication, faster handoffs, lower review cost, or better integration. These remain falsifiable prospective outcomes.
